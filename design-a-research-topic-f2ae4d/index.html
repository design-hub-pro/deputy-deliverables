Research Topic Design

3 LLM Research Topics for a Strong Middle Schooler

Each topic asks a measurable question, runs on a laptop with a hosted LLM API (a few dollars, no GPUs), and plays to this student's specific strengths.

🏆 Strong in USACO 🌐 Built a web app ⏱️ ~6 weekends 💵 A few dollars of API

What makes these good & feasible

  • Feasible for a middle schooler: no model training, no big datasets — just a few hundred API calls to Claude / GPT / Gemini, or a free local model via Ollama.
  • Real research, not a demo: each one produces a number, a chart, and a conclusion someone could argue with. That's the line between "research" and "I built a chatbot."
  • Plays to their strengths: USACO skill → writing an automated grader & reasoning about algorithms; web-app skill → shipping an interactive results dashboard that makes any project look finished.

1Can LLMs Solve USACO?

Measure how good today's LLMs really are at competitive programming — and carefully categorize why they fail.

Why it fits: The student already knows USACO deeply. Most researchers benchmarking code LLMs can't tell a "wrong algorithm" mistake from an "off-by-one" mistake. This student can — that domain expertise is the whole edge.

Research questions

  • What fraction of Bronze / Silver / Gold problems can an LLM solve correctly?
  • Where does it break? Misreads the problem, wrong algorithm choice, right algorithm but bad edge cases, or TLE from bad complexity?
  • Does the intended solution's complexity predict failure better than problem length?

What to build

  • A dataset of ~40–60 public USACO problems with official test data.
  • An automated judge harness (the USACO-flavored part): LLM → extract code → compile → run against official tests → record which tests failed.
  • A hand-built failure taxonomy — the human-labeled analysis is the real scientific contribution.
  • A web dashboard: accuracy by division + a heatmap of error-type vs. difficulty, with clickable examples.

Experiments

  • Baseline accuracy per division.
  • Does chain-of-thought ("explain the algorithm first, then code") help?
  • Does giving the algorithm tag ("this is DP") close the gap — separating "can't pick the algorithm" from "can't implement it"?

2Does "Self-Correction" Actually Help?

LLMs are often told to "check your work." Does that make them more correct — or just more confident?

Why it fits: A clean, controlled experiment — exactly the rigorous, don't-fool-yourself thinking a USACO student is good at. Minimal building, maximum scientific care.

Research questions

  • On checkable tasks (math word problems, logic puzzles), does asking the model to review & revise improve accuracy?
  • Does self-correction ever turn a right answer wrong? What's the net effect?
  • Does the model actually know when it's wrong (is its confidence calibrated)?

What to build

  • One dataset with auto-checkable answers (e.g. GSM8K math, or logic puzzles the student writes a checker for — very USACO).
  • Three conditions: Direct (answer once) · Self-critique (answer, then revise) · Self-consistency (sample 5×, majority vote).
  • An automated correctness checker + a dashboard showing right→wrong vs. wrong→right transitions.

Expected finding

An honest number: "self-correction helps +N% on task A but hurts on task B, and majority-voting beats self-critique." Self-correction often helps less than people assume — which is genuinely interesting.

3Small-Scale RAG: Does Retrieval Cut Hallucination?

Build a "closed-book vs. open-book" LLM over a chosen document set, and measure how much giving it the source reduces made-up answers.

Why it fits: This leans on the web-app skill. RAG powers most real LLM products, so building one is resume-worthy — and the fixed question set keeps it honest as research, not just a demo.

Research questions

  • For questions about one corpus (a game's rules, a Wikipedia subset, Pokédex data, a school handbook), how often does the LLM hallucinate with vs. without retrieval?
  • How does the number of retrieved chunks (k = 1, 3, 5, 10) affect accuracy? Is there a point where more context hurts?
  • Does chunk size matter?

What to build

  • A self-contained corpus + ~50 questions with known correct answers (ground truth).
  • A simple RAG pipeline: chunk → embed → retrieve top-k → answer using only those chunks.
  • A web app where you type a question and see both the retrieved passages and the answer — great for demos and debugging.
  • An eval script scoring answers vs. ground truth across every k setting.

Expected finding

"Without retrieval the model is right X% and confidently wrong Y%; with k=3 it's right Z%; beyond k=5 accuracy plateaus and irrelevant chunks start causing errors."

⭐ Recommended: Topic 1

It's the most differentiated. Very few people can do a careful failure analysis of LLMs on USACO problems, because it needs exactly this student's rare combo — competitive-programming expertise + coding a grader + building a dashboard. It's the topic where this specific student has an unfair advantage.

A realistic 6-weekend plan (using Topic 1)

  1. Weekend 1 — Collect 15–20 problems + test data; get one LLM→code→compile→judge loop working end to end.
  2. Weekend 2 — Scale to 40–60 problems; automate the batch; store results in JSON/CSV.
  3. Weekend 3 — Hand-label failure types on the first batch; refine the taxonomy.
  4. Weekend 4 — Run the prompting experiments (chain-of-thought, algorithm hints).
  5. Weekend 5 — Build the results dashboard (charts + clickable examples).
  6. Weekend 6 — Write the 4–6 page report; add one stretch goal.

Keeping it "research"

  • Fix your dataset & questions before looking at results — no cherry-picking.
  • Always report a baseline to compare against.
  • Report what didn't work — negative results are real results.
  • Keep every prompt & script in a repo so it's reproducible.
